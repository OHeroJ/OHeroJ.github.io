算法的时间复杂度

发表于 2018-04-02 | | 阅读次数:

本篇仅为记录所用，方便查阅

  • O(1) 常量型，简单语句
  • O(lg(n)) 对数型， 二分法查找
  • O(n) 线性型 （顺序查找）
  • O(nlg(n)) 比线性差，(快速排序、堆排的平均时间)
  • O(n^2) 选择和插入
  • O(n^3) 立方形 2n*n 矩阵相乘
  • O(C^n) 指数型 (旅行商问题，集合划分)

估算基本算法的阶

  • 简单循环, n 线性增加
  • 嵌套循环 m * n,排序算法一般 O(n^2)
  • 二分法 一般是 O(lg(n))
  • 分而治之 O(nln(n))
  • 组合 阶层

注重实效的途径

  • DRY

    • 发生的时机
      • 强加的重复，开发者觉得他们无可选择，环境似乎要求重复
      • 无意义的重复，没有意识到重复
      • 无耐心的重复，偷懒
      • 开发者之间的重复，统一团队，多个人重复了同样的信息

  • 正交性

    如果两条直线相交成直角，它们是正交的。两条直线互不依赖

    设计良好的系统，数据库与用户界面是正交的： 你可以改动界面，而不影响数据库，更改数据库，而不用改动界面。

    • 正交的好处

      消除无关事物之间的影响

      要想设计自足的组件： 独立，就有单一，良好定义的目的
      提高生产率与降低风险

      • 提高生产率：
        改动得以局部化
      • 促进复用
      • 降低风险

    • 编码

      • 让你的代码保持解耦。
      • 避免使用全局函数
      • 避免写相似函数

    • 测试

    • 文档
  • 制作原型
    • 制作原型架构
      • 主要的组件的责任是否起到了良好的定义？是否适当？
      • 主要组件的协作是否得到了良好的定义？
      • 耦合是否得到以最小化？
      • 能否确定重复的潜在来源？
      • 接口定义和各项约束是否可接受
      • 每个模块在执行过程中是否能够访问到其所需的数据？是否能在需要时进行访问

Swift 设计模式之策略模式

发表于 2018-03-01 | | 阅读次数:

策略模式(Strategy Pattern): 定义一系列算法类，将每一个算法封装起来，并让它们可以相互替换，策略模式让算法独立于使用它的客户而变化，也称为政策模式(Policy)。策略模式是一种对象行为型模式。
策略模式提供了一种可插入式(Pluggable)算法的实现方案

案例

Sunny软件公司为某电影院开发了一套影院售票系统，在该系统中需要为不同类型的用户提供不同的电影票打折方式，具体打折方案如下：
(1) 学生凭学生证可享受票价8折优惠；
(2) 年龄在10周岁及以下的儿童可享受每张票减免10元的优惠（原始票价需大于等于20元）；
(3) 影院VIP用户除享受票价半价优惠外还可进行积分，积分累计到一定额度可换取电影院赠送的奖品。
该系统在将来可能还要根据需要引入新的打折方式。

优点

  1. 对“开闭原则”的完美支持，可以在不修改原有系统的基础上选择算法或行为，也可以灵活地增加新的算法或行为
  2. 提供了一种可以替换继承关系的办法。
  3. 提供了一种算法的复用机制

缺点

  1. 产生很多策略类
  2. 策略模式只适用于客户端知道所有的算法或行为的情况
